1. Introduction


1.1 The Purpose of the Document

The purpose of this document will be to outline and describe the application, the application interface requirements, design requirements, business and marketing plan and finally the implementation process.


1.2 Scope of the Document

Our application will be web based. It will require that a user will have the ability to; use or locate a computer, connect to the internet (Wi-Fi or wired ethernet connections), and use a web browser to navigate to our application site.


1.3 Overview of the Application

This section describes the software requirements for the Beer Buddy application. The purpose of our application will be to make (preferred) beer acquisition easier and faster. The application aims to increase the speed of the buying process. Presently, customers go through a six-step process when purchasing anything. In order; problem recognition, information search, evaluation of alternatives, purchase decision, purchase and post purchase evaluation. Essentially the problems a consumer faces are; they don’t know what they want, they must work and evaluate choices (costly endeavor) and finally they are left to review themselves whether they like the products/services they purchased. Our web application will allow users to use essential primary criteria to make choices that will best suit their preferences. Since the consumer already has a historic previous preference they will be able to make the choices that will offer them relevant options to choose from instead of testing out every product to determine if they enjoy it or not.



2. Business Context


2.1 Ease of Use

This section describes the consumer view for the Beer Buddy application. Initially the user will have the main screen with two options. An option where they choose whether they are familiar with beers or they are novice beer consumers. The first choice will then lead to a new page that will be selection criteria. After the user chooses their preferences, the application will generate a list of relevant beers. The user will then be able to select the beer they would like, and it will be guaranteed to be of their liking (such that they entered preferences they actually enjoyed). The latter option of novice consumers will lead to a page which will have approximately five to ten beers which are extremely different. The user will be able to select the one that interests them, the application will list the qualities or key characteristics of the drink and any relevant reviews. For all the drinks the assumption will be made that they are available for purchase locally (for this project local is assumed to be Waterloo Region).


2.2 Scalability

This section describes the scalability for the Beer Buddy application. Initially, the application is designed to be accessible via websites. Anyone with a computer and access to a browser will be able to access it and then embark on a trip to purchase the drink that best fits their criteria. As a business model, this application would ideally made for tablets which would be available at local bars, pubs, and nightclubs. This would make it easier for people to pick out new drinks or similar drinks to try out while enjoying their night out with friends or their partners. If implemented in tablet mode, the application could also be used in local beer stores or stores with an assortment of beers/alcohols that are available for purchase. This would increase the stores revenue and the data can then be recorded and sold to beer/alcohol manufacturers so that they can monitor what the consumers like to drink and what not.



3. Current System Description


3.1 Purpose

The goal of Beer Buddy is to provide insight and advice in the beer choices of its users. They can enter criteria that they want to use to query related beers, as well as adding custom beers to the database (would use the API). The service assumes that the beers are available locally (same city as one resides in).


3.2 User Characteristics

User must understand terms about beer, and have basic aptitude towards search engines.



Select Criteria

Option to choose whether they are a novice consumer of beer or not

Pick Criteria Attributes

Pick preferences of what kind of beer they like or enjoy. What combination of these do they prefer.

View Queried Beers

A list of relevant beers based on the attributes selected (along with key information about local availability, tags, ratings (if applicable)

Add Custom Beer

If preferred beer not found, or user has already tried all the listed suggested beers. A window will be presented (through an “Add New Beer” button) for the user to enter a new beer that fits the selection criteria previously listed (if not selected the user will enter attributes on their own). Then the beer will be submitted for review by moderators who will determine the legitimacy of the submission and whether it is accurate with the attributes listed as its traits.



4. Functional Requirements


4.1 Purpose

  1. Client enters information about beer through tags/search filters
  2. Produce search results
  3. Search results will be sorted based on options entered by user or default options

4.2 Reviews

  1. Client posts opinions of beers
  2. Client rates beers
  3. Forum page for each product for discussion

4.3 Client data-entry

  1. System for users to enter their own beers through popular vote, etc. (TBD)
  2. Client entries will have information about the user

4.4 Data Storage

  1. Heroku database hooked up to the Django REST Framework (API)


5. Interface Requirements


5.1 Selection Screens (web-based)

Users select the criteria they want, adjust the attributes of that property, and then choose to select more criteria or query the database. These screens will have intuitive selective features such as; drop down menus, check boxes. The need for a keyboard will be minimal, the only time it should be used is if the user wants to submit a review or if the user wants to add a custom beer.



6. Performance Requirements


6.1 Hardware Requirements

This application requires a computer or smartphone that connects with internet and must have an internet browser.


6.2 Software Requirements

This application will require that the device have an ability to connect to the internet and have uncorrupt version of an internet browser.


6.3 Human Capability Requirement

This application will require that the user be able to read the instructions provided and understand beer tastes and qualities. The consumer is expected to have tried and be aware of the different flavours/renders unless they are a novice and choose the novice option.



7. Design Requirements

10. Portability


  • The service needs to be accessible on all browsers (Chrome, Firefox, Microsoft Edge, Opera, Safari, Internet Explorer (if time permits), etc). Since the application will be Bootstrapped, mobile responsiveness will be built-in with the app. The service will be responsive to all current versions of web browsers.


11. Reusability


  • The application is assumed to be open source, available for public use. Any programmer will be allowed to reuse part or all the code during the preliminary development stage. This group reserves the right to refuse the open source availability of the application code.


12. Optional Features


  • Databases can be updated with information from (local) beer vendors with price, availability, sales, etc.
  • The service assumes that the user will have some knowledge beers and an understanding of what they like. If however a beer-incompetent individual wants to use the service, we will implement a help feature button which explains the significantly different beers so that the users have a chance to pick an “extreme” option and then decide what they like.
  • Review feature, time permitting this will be attempted so that beer consumers can rank their personal favorites.
  • Future scaling option to create login mechanism. for reviewing purposes. Users of the application will be able to have an account that will store all their pending or approved purchases.
